RELATORIO DO PROJETO DE TCC
KnowledgeGraphModelSchoolEducation

1. PROPOSITO DO PROJETO

Este projeto foi concebido para investigar como dados escolares podem ser utilizados para antecipar problemas academicos e apoiar a tomada de decisao pedagogica. A ideia central e prever o desempenho futuro de alunos a partir de sinais ja observados no historico escolar, com o objetivo de permitir intervencoes antes que haja reprovacao ou queda severa de desempenho.

Desde o inicio, a intencao do trabalho foi ir alem de simplesmente classificar o aluno como aprovado ou reprovado. O objetivo academico e tecnico e entender se ha padroes consistentes em notas, faltas, pagamentos e outras variaveis escolares que possam indicar risco futuro e, com isso, fornecer subsidios para professores e escola ajustarem a abordagem com antecedencia.

2. IDEIA INICIAL DO TCC

A ideia inicial do projeto foi modelar o contexto escolar como um grafo de relacoes. Nessa visao, alunos, matriculas, notas, faltas, pagamentos e responsaveis poderiam ser representados como nos e arestas, permitindo o uso de tecnicas de grafos e, potencialmente, redes neurais para grafos.

Essa proposta fazia sentido como exploracao inicial porque o dominio escolar contem relacoes entre entidades diferentes, como:
- aluno e matricula
- matricula e notas
- matricula e faltas
- matricula e pagamentos
- aluno e responsaveis

O intuito dessa fase era verificar se uma estrutura relacional mais rica ajudaria na identificacao de padroes de risco, dependencia ou comportamento academico.

3. CAMINHO TRACADO ATE AGORA

O arquivo main.py demonstra a sequencia historica de experimentos e funcoes do projeto, da fase orientada a grafo ate a fase atual com Random Forest. Em ordem crescente, o caminho observado no codigo foi o seguinte:

3.1. Estruturacao inicial do projeto

Foi criada uma base organizacional com:
- conexao com a origem de dados
- extracao para arquivos CSV
- classe central de gerenciamento com GraphManager
- separacao em dois blocos de trabalho principais:
  BDBasic
  BDTreatment

O bloco BDBasic concentra a fase de exploracao com grafos, testes comparativos e LSTM. O bloco BDTreatment concentra uma tentativa posterior de tratar os dados de forma tabular para alimentar um Random Forest.

3.2. Extracao de dados para CSV

Na fase inicial, foram extraidos arquivos tabulares com dados de:
- alunos
- medias/notas
- pagamentos
- faltas
- responsaveis

Esses arquivos ficaram registrados em:
- BDBasic/Result/CSV/aluno.csv
- BDBasic/Result/CSV/media_nota_aluno.csv
- BDBasic/Result/CSV/pagamento_aluno.csv
- BDBasic/Result/CSV/faltas_aluno.csv
- BDBasic/Result/CSV/responsaveis_aluno.csv

Essa etapa foi importante porque criou a primeira base concreta para analise e experimentacao.

3.3. Analises exploratorias iniciais

No fluxo do main.py aparecem as seguintes etapas exploratorias:
- analise de correlacao
- comparacao de classificadores tradicionais
- comparacao de classificadores com PCA

Essa fase indica que o projeto tentou medir a estrutura estatistica dos dados antes de partir para modelos mais complexos.

3.4. Modelagem orientada a grafo

Depois da extracao e da exploracao inicial, o projeto passou a:
- montar um grafo heterogeneo a partir dos CSVs
- visualizar o grafo
- consultar estudantes no grafo com networkx
- consultar estudantes no grafo em estrutura preparada para PyTorch

Como resultado dessa linha de trabalho, foram gerados artefatos como:
- BDBasic/Result/Preprocessing/networkx_graph.graphml
- BDBasic/Result/Preprocessing/pytorch_graph.pt

Essa fase foi coerente com a proposta original do TCC e serviu para validar a possibilidade de representar o problema como rede de relacoes.

3.5. Tentativa com LSTM

Em seguida, o projeto incorporou uma etapa de treinamento com LSTM para previsao de notas. O proprio main.py registra uma lista de itens planejados para esse modelo, incluindo:
- processamento de atributos
- normalizacao
- padronizacao de sequencias
- embeddings
- enriquecimento de atributos
- camadas recorrentes
- camadas de ativacao
- regularizacao
- treinamento e avaliacao

Essa fase mostra uma tentativa de tratar o problema como sequencia temporal de desempenho, o que conceitualmente se aproxima bem do dominio escolar.

3.6. Migracao para uma abordagem tabular com Random Forest

Em um momento posterior, o projeto mudou de direcao pratica e passou a testar uma abordagem tabular com Random Forest. A justificativa implicita no codigo e no estado atual do repositorio e que os dados observados apresentavam padroes mais regulares e menos dependentes de estrutura de grafo do que se imaginava inicialmente.

Nessa nova fase, o projeto:
- transformou dados de notas em um CSV de linha unica por aluno
- criou funcoes de entendimento dos dados
- montou um fluxo de preprocessamento
- treinou e avaliou um modelo Random Forest
- salvou artefatos do modelo treinado

Arquivos dessa fase:
- BDTreatment/Result/media_nota_aluno_single_line.csv
- BDTreatment/Result/data_understanding_report.txt
- BDTreatment/Result/model_evaluation_report.txt
- BDTreatment/Result/random_forest_model.pkl

5. ESTADO ATUAL DO PROJETO

Hoje o projeto se encontra em uma transicao entre duas visoes:

- a visao inicial, mais academica e relacional, baseada em grafos e sequencias
- a visao mais pragmatica, baseada em modelagem tabular supervisionada

O repositorio mostra que a implementacao atual mais funcional esta na linha do Random Forest. No entanto, a arquitetura ainda nao representa uma pipeline real de previsao escolar pronta para sustentar o resultado final do TCC.

6. LIMITACOES IDENTIFICADAS NO ESTADO ATUAL

As principais limitacoes observadas na analise do projeto foram:

6.1. Base pivotada em excesso

O arquivo de linha unica por aluno ficou muito largo e esparso, reunindo muitas colunas para poucos registros relativos por estudante. Isso dificulta a generalizacao e nao representa bem a natureza temporal dos dados.

6.2. Perda de granularidade temporal

A forma atual simplifica demais a ordem natural dos eventos escolares. O problema real e temporal: cada previsao deveria usar apenas o que estava disponivel ate aquele momento.

6.3. Uso incompleto das fontes de dados

Embora o projeto possua dados de faltas e pagamentos, a modelagem atual gira principalmente em torno das notas. Ainda nao existe uma integracao estruturada e segura dessas fontes em uma unica base de modelagem.

6.4. Reducao excessiva da nota em classes

Em parte do fluxo atual, a nota numerica e reduzida a classes categorizadas. Isso pode ser util em algumas analises, mas enfraquece a previsao fina de desempenho.

6.5. Falta de pipeline reproduzivel de ponta a ponta

O repositorio ainda esta organizado em blocos de experimento. Falta uma arquitetura unica e clara para:
- extrair
- limpar
- integrar
- gerar atributos
- treinar
- validar
- prever
- gerar alertas

6.6. Falta de criterio final de avaliacao alinhado ao problema real

O modelo precisa ser avaliado por metricas que facam sentido para a escola e para o TCC, como:
- erro medio de previsao de nota
- capacidade de detectar alunos em risco
- interpretabilidade da previsao
- estabilidade temporal

7. APRENDIZADO OBTIDO COM O CAMINHO JA PERCORRIDO

O caminho tracado ate aqui foi valido e importante para o TCC porque permitiu:

- testar hipoteses diferentes
- entender melhor a estrutura do problema
- verificar que a abordagem em grafo, embora interessante, nao e necessariamente a mais eficiente como primeira solucao operacional
- perceber que o problema tem forte natureza temporal e tabular
- confirmar que ha material suficiente para evoluir para uma pipeline mais madura

Em outras palavras, o projeto nao esta perdido nem recomeçando do zero. Ele passou por uma fase de descoberta e agora tem condicoes reais de consolidacao.

8. CONTINUACAO RECOMENDADA PARA CHEGAR A UM FIM DESEJAVEL NO TCC

Para chegar a um resultado final forte, coerente e defensavel academicamente, a continuacao recomendada e a seguinte:

8.1. Refatorar a arquitetura do projeto

O primeiro passo e reorganizar o projeto em uma pipeline clara, separando:
- ingestao de dados
- tratamento
- integracao entre fontes
- engenharia de atributos
- treino
- avaliacao
- inferencia
- geracao de relatorios

8.2. Abandonar a base extremamente pivotada como base principal de treino

O formato principal deve passar a ser uma base longitudinal, em que cada linha represente um evento ou observacao relevante do aluno em determinado momento.

Exemplo conceitual:
- aluno
- ano letivo
- serie
- disciplina
- etapa atual
- historico disponivel ate esse ponto
- alvo: proxima nota

8.3. Integrar notas, faltas e pagamentos de forma temporal

Cada observacao do modelo deve conter apenas informacoes disponiveis ate o momento da previsao. Assim, sera possivel avaliar:
- efeito das faltas recentes
- efeito de atrasos ou irregularidades financeiras
- tendencia do aluno por disciplina
- historico global de desempenho

8.4. Criar atributos realmente informativos

Entre os atributos desejaveis:
- ultima nota
- media das ultimas notas
- tendencia de melhora ou piora
- quantidade de faltas na disciplina
- quantidade de faltas recentes
- media geral do aluno
- desempenho por area do conhecimento
- atrasos de pagamento
- variabilidade das notas
- historico do ano anterior

8.5. Definir dois objetivos de modelagem

O projeto pode ficar mais forte com dois alvos complementares:

- regressao: prever a proxima nota numerica
- classificacao de risco: prever se o aluno entrara em zona de alerta pedagogico

8.6. Estabelecer uma validacao temporal correta

O treino nao deve depender apenas de divisao aleatoria. O ideal e validar por janela temporal, por exemplo:
- treino com anos anteriores
- validacao em um ano intermediario
- teste no ano mais recente

Isso fortalece a defesa do TCC e aproxima a avaliacao do uso real.

8.7. Comparar modelos simples e modelos mais fortes

O Random Forest pode permanecer como baseline, mas a continuacao desejavel inclui comparar com:
- baseline ingenuo, como repetir a ultima nota
- media movel das ultimas notas
- Gradient Boosting
- CatBoost
- LightGBM, se aplicavel

8.8. Gerar uma camada de interpretacao para o professor

O resultado final nao deve ser apenas uma nota prevista. O ideal e produzir tambem:
- nivel de risco
- fatores que mais influenciaram a previsao
- sugestao de atencao pedagogica

Assim, o TCC se fortalece como ferramenta de apoio real.

9. FIM DESEJAVEL PARA O TCC

O fim desejavel para este TCC nao e apenas apresentar um modelo treinado. O resultado ideal e entregar:

- uma pipeline clara e reproduzivel
- um problema bem formulado
- uma base integrada e consistente
- um modelo que preve a proxima nota com qualidade aceitavel
- um modulo de identificacao de risco academico
- relatorios compreensiveis para uso humano
- uma justificativa tecnica solida para a escolha da abordagem final

Se isso for atingido, o TCC tera valor academico e aplicacao pratica.

10. PROXIMO PASSO IMEDIATO

O proximo passo recomendado, a partir deste ponto, e iniciar a refatoracao do projeto e da arquitetura para transformar os experimentos atuais em uma pipeline real. Essa refatoracao deve respeitar o historico do projeto, aproveitar o que ja foi construido e reorganizar o repositorio para a fase final do TCC.
